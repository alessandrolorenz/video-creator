This plan is documentary work only. Do not run `git init`, install dependencies, or create production source until:

1. M0.0 is explicitly approved and frozen ‚Äî **complete**;
2. the corrected M0.1 spec is marked `Approved / Frozen` ‚Äî **complete**;

4. the corrected plan receives an independent `PASS` or accepted `PASS WITH NOTES` ‚Äî **complete: PASS WITH NOTES**;
5. the product owner explicitly authorizes the named checkpoint ‚Äî **Checkpoints 0‚Äì5 complete**.

Implementation stops after each authorized checkpoint. Evidence is recorded before the next checkpoint is requested.

## Checkpoint completion protocol

Checkpoint 0 uses its explicit two-commit baseline/evidence sequence. For Checkpoints 1‚Äì5, authorization of the checkpoint includes one local checkpoint commit and no push:

1. implement only the authorized files;
2. run the checkpoint verification;
3. update the implementation report with commands/results, without attempting to record the commit's own future SHA;
4. create one local checkpoint commit containing the authorized work and its evidence;
5. confirm a clean worktree;
6. report the resulting commit SHA in the external checkpoint handoff;
7. request authorization for the next checkpoint.

This makes the Git `HEAD` after Checkpoint 4 an immutable candidate containing Checkpoints 0‚Äì4. A rollback is performed only with explicit authorization and reverts the complete checkpoint commit‚Äîincluding its implementation-report evidence‚Äîwhile preserving earlier checkpoints.

## Dependency decisions

### Runtime dependencies

- `react` and `react-dom`: renderer only.
- one small runtime schema validator (planned: `zod`): shared IPC parsing; provider SDK types are forbidden.

### Development dependencies

- `electron`: desktop runtime for development and smoke; no installer tooling in M0.1.
- `vite` and the official React plugin: renderer build only.
- `typescript` aliased to the official TypeScript 6 compatibility package, plus Node/React type packages: strict application and package checking with a programmatic API supported by `typescript-eslint`.
- `vitest`: unit and architecture tests.
- ESLint with TypeScript/React-Hooks support and Prettier: static/style checks.

No FFmpeg wrapper, OpenAI/AI SDK, SQLite/ORM, exporter, Electron Forge/Builder, auto-updater, telemetry, state framework, router, component library, or native addon is permitted in M0.1.

## Security plan

- Export window options and navigation policy as testable pure configuration/functions.
- Keep Electron imports out of renderer code and out of framework-independent packages.
- Parse unknown IPC payloads into trusted values; return typed error envelopes for invalid input.
- Freeze the exposed preload surface and expose no raw `ipcRenderer` object.
- Keep secrets and environment data entirely outside renderer declarations.
- Load only the packaged local renderer entry; fail closed if the entry is missing.
- Use a restrictive CSP and deny new-window/navigation attempts.
- Add architecture tests that inspect workspace manifests plus static ESM imports, dynamic `import()`, and CommonJS `require()` for forbidden edges and dangerous bridge names.

## Test-first order

1. Write failing domain invariant tests, then implement domain constructors/helpers.
2. Write failing timeline schema/round-trip tests, then implement the minimal timeline contract.
3. Write failing architecture-boundary fixtures, then implement the checker.
4. Write failing Electron option/navigation/IPC tests, then implement main and preload code.
5. Write the renderer output test, then implement the neutral screen.
6. Add the CI workflow only after every local command is reproducible.

## Risks, rollback, and stopping points

| Risk | Mitigation |
|---|---|
| Starting from no Git history | Checkpoint 0 creates a preserved documentary baseline before source work. |
| Electron/preload module mismatch | Use an explicit sandbox-compatible CommonJS preload output and test its resolved path. |
| Security settings regress during refactor | Export/test window options and reject navigation independently of app launch. |
| Runtime validation exists only in types | Handlers accept `unknown` and must parse before work; invalid-input tests are required. |
| Workspace boundaries become conventions only | Executable import guard runs locally and in CI. |
| Toolchain churn | Pin exact lockfile versions after compatibility verification; avoid prereleases. |
| Scope grows into product functionality | Each checkpoint has an explicit file boundary and forbidden-dependency check. |

Every checkpoint ends with verification and a stop. No destructive reset, force operation, remote push, or publication is part of this plan. The only planned deletion is the explicitly authorized rollback of newly created `.git` metadata or workspace-local generated artifacts; original documents are never deleted.
