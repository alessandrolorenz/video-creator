# M0.1 Amendment 001 — Repository Fixture Policy

Status: APPROVED / APPLIED — 2026-07-21

Prepared: 2026-07-21  
Affected frozen specification: `M0.1-repository-foundation-spec.md`

## Reason

The frozen M0.1 roadmap requires a fixture policy, and the frozen product principles require fixtures to be small, redistributable, and versioned. The M0.1 foundation specification omitted a concrete fixture-policy deliverable. The third independent plan judge classified this as underdelivery.

## Old rule

The frozen M0.1 foundation specification contains no fixture-policy item in scope and no corresponding verification requirement.

## Proposed new rule

Add the following in-scope requirement:

> A repository fixture policy at `docs/fixture-policy.md` defining approved locations, provenance and license records, size budgets, versioning/checksums, privacy restrictions, synthetic-data preference, review requirements for binary media, and milestone ownership. M0.1 creates policy only and adds no binary media fixture.

Add the following verification requirement:

> `docs/fixture-policy.md` exists, is linked from the root README, contains every required policy section, and M0.1 contains no binary media fixture.

## Required policy contents

The implementation checkpoint must create `docs/fixture-policy.md` with these rules:

1. future fixture sources live under `packages/fixtures/` only after a milestone explicitly authorizes them;
2. synthetic fixtures are preferred;
3. every fixture records origin, author/owner, license, intended tests, creation command when synthetic, size, and SHA-256;
4. personal, confidential, biometric-identifying, or otherwise sensitive footage is forbidden;
5. committed fixtures must be minimal and redistributable;
6. binary and total-repository size budgets require explicit values before the first media fixture is added;
7. replacements create new versioned files/checksums rather than silently changing golden inputs;
8. Git LFS is not introduced without a later approved decision;
9. M0.1 adds no audio, video, image, subtitle, transcript-data, archive, or other binary fixture;
10. fixture exceptions require a documented milestone amendment and review.

## Affected artifacts

- `docs/specs/M0.1-repository-foundation-spec.md` — amended only after approval;
- `docs/specs/M0.1-implementation-plan.md` — maps the policy to Checkpoint 1;
- `README.md` — receives a policy link during Checkpoint 4;
- `docs/fixture-policy.md` — created during authorized implementation;
- CI/architecture evidence — confirms no M0.1 binary fixture.

## Affected tests and verification

- Checkpoint 1 verifies the policy file and all required headings/rules.
- The architecture/repository guard rejects binary fixture extensions during M0.1.
- Checkpoint 4 links the policy from README and runs the guard in CI.
- The implementation judge verifies that no binary media fixture or Git LFS configuration was introduced.

## Scope and non-goals

This amendment adds documentation and a repository guard rule only. It does not authorize fixture binaries, sample media generation, FFmpeg, transcripts, ingest, test-media downloads, Git LFS, or product functionality.
