## Architecture findings

### Reusable foundations

- The Electron main/preload/renderer separation is real and security-tested.
- The main process can own native file dialogs and privileged paths without exposing them to the renderer.
- Strict runtime-validated request/response envelopes already exist and can be extended narrowly.
- `TimeUs` and `SourceRange` provide the canonical time foundation.
- The media job contract already models cancellation, progress, success, failure, and typed errors.
- CI and the architecture/fixture guard provide a strong baseline for incremental M1.0 checks.

### Required evolution

- Add a pure `packages/transcript` workspace that depends only on domain types/runtime validation.
- Add media-asset and probe-result contracts without moving Node/Electron/process APIs into framework-independent packages.
- Introduce one dedicated privileged media-probe worker boundary. The renderer must never receive a path, executable, argument list, raw process handle, filesystem API, or raw IPC object.
- Evolve the M0.1 guard from milestone-specific prohibitions into M1.0 rules: continue forbidding Node/framework imports in pure packages while allowing a specifically owned worker/main adapter to use the minimum Node process APIs.
- Replace the neutral M0.1 screen with a narrow ingest/status UI only after implementation is separately authorized.

## External-tool decision proposed for Gate 2

M1.0 development will use a declared external `ffprobe` executable. It will not download, bundle, package, sign, or redistribute FFmpeg. Distribution and license packaging remain M4 work under the frozen risk register.

Proposed resolution order is main-process-only configuration followed by inherited `PATH`; neither executable location nor environment is renderer-controlled or renderer-visible. A missing executable is an explicit user-visible prerequisite error, not an automatic download.

The process contract must use a fixed executable and fixed argument template, `shell: false`, bounded stdout/stderr, timeout, cancellation, and exit/parse validation.

## Transcript-contract decisions proposed for Gate 2

- One versioned JSON input contract with `word` or `segment` granularity.
- Explicit input time unit: microseconds, milliseconds, or seconds.
- Deterministic nearest-microsecond conversion at the boundary; canonical state uses only safe-integer `TimeUs`.
- Input order is authoritative and must already be monotonic; M1.0 never silently sorts or merges entries.
- V1 rejects overlapping timed entries and entries outside the imported asset duration.
- Text is preserved; matching normalization belongs to M1.1.
- The selected transcript file is bound by the main process to the active asset; an input file cannot select another asset or path.
- File size, entry count, string length, and parsed-object limits are mandatory denial-of-service controls.

## Fixture-policy decision

M1.0 requires no committed binary fixture and no fixture-policy amendment:

- unit/integration tests use inline or temporary text JSON and captured minimal `ffprobe` JSON;
- process tests use injected fakes or a fake executable created in a test temporary directory;
- CI does not download media;
- manual smoke uses an explicitly selected local media file and transcript outside the repository;
- no media, screenshot, archive, or generated binary is committed.

If a later checkpoint needs committed media, it must first approve an amendment with per-file and aggregate budgets, provenance, license, privacy, digest, and purpose.

## Security and privacy constraints

- Native file selection occurs only in Electron main and is attached to the current window.
- Main resolves and validates selected files as readable regular files.
- Absolute paths remain in privileged state and are redacted from renderer responses and ordinary error messages.
- Renderer receives only opaque IDs, display names, sanitized metadata, job state, and typed error codes.
- IPC and worker messages enter as `unknown` and are strictly parsed before filesystem or process work.
- No generic read-file, choose-path, execute, shell, environment, or arbitrary-argument bridge is permitted.
- Source media is never copied, modified, transcoded, uploaded, or hashed in full during M1.0.
- Transcript content remains local and is not sent to any AI provider.

## Product and UX boundary

Required visible states are empty, choosing file, probing/validating, ready, cancelled, and actionable error. The ready state shows only media/transcript summaries and replacement actions.

M1.0 must not expose passage selection, matching confidence, ambiguity candidates, timeline, playback, trim/slip/reorder, rendering, export, project persistence, AI, multiple assets, or packaging.

## Current official reference check

- [`ffprobe` documentation](https://ffmpeg.org/ffprobe.html): machine-readable media/stream inspection and positive failure exit for unreadable or unrecognized input.
- [Electron `dialog`](https://www.electronjs.org/docs/latest/api/dialog): native file selection belongs to the main process and supports a parent window and file filters.
- [Electron `utilityProcess`](https://www.electronjs.org/docs/latest/api/utility-process): Node-enabled utility process with main-process ownership and message ports.
- [Node child-process documentation](https://nodejs.org/api/child_process.html): asynchronous spawn, AbortSignal cancellation, close/error handling, and the security warning against shell execution with unsanitized input.

These references validate feasibility; exact versions remain pinned by the repository lockfile and the future approved plan.
