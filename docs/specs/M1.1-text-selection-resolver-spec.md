Prepared: 2026-07-22 (America/Sao_Paulo)
Process: lean risk-tiered gates ([`../07-sdd-workflow-and-gates.md`](../07-sdd-workflow-and-gates.md), [`../decisions/ADR-004-lean-risk-tiered-gates.md`](../decisions/ADR-004-lean-risk-tiered-gates.md))

## Objective

Given the canonical timed transcript produced by M1.0 and a set of selected text passages, resolve each passage to the exact source time range(s) it covers — deterministically, with no media, no `ffprobe`, and no I/O. This is the heart of the product: turning "the words I want" into "the timestamps to cut." It can be fully proven by pure and property-based tests before any UI or render exists.

## User outcome

As a creator, I can provide the passages I want to keep (in reading order) and get back, for each passage: the source time range, whether the match was exact or fuzzy, a confidence signal, and — when the same words occur more than once — the set of candidate locations so a later step (or I) can disambiguate. Passages that cannot be matched are reported clearly rather than silently dropped.

## In scope

- A new **pure, framework-independent** workspace package (proposed name `selection`) depending only on `domain` and `transcript`.
- Deterministic text normalization for matching (case folding, whitespace collapsing, punctuation and diacritic policy) applied identically to passages and transcript text.
- **Exact ordered matching**: a passage matched to a contiguous span of transcript entries, yielding one `SourceRange` (from the first entry's `startUs` to the last entry's `endUs`).
- **Fuzzy ordered matching**: bounded, deterministic tolerance (token-level and/or bounded edit distance) for punctuation/case/minor-wording drift, never crossing entry-ordering.
- **Ambiguity detection**: when a passage's normalized text occurs at more than one location, return all candidate ranges rather than guessing.
- **Alternatives**: for a chosen/most-likely match, expose the other candidate ranges.
- A typed **resolution result**: per passage → `matched` (exact/fuzzy, one `SourceRange`, confidence, alternatives) | `ambiguous` (ordered candidate ranges) | `unmatched` (reason).
- Pure unit tests and property-based tests (no negative/zero durations, ranges within transcript bounds, stable ordering, normalization idempotence, round-trip stability).

## Explicit non-goals

- Timeline construction, clip ordering, handles, preview, playback, trim, slip, reorder (M1.2).
- Rendering, transcoding, MP4 output, XML/EDL/OTIO export, NLE interop (M1.3).
- Any renderer/preload/IPC wiring, native paste UI, or highlighting — M1.1 is engine-only. A minimal UI to exercise it is a **separate** authorized follow-on, not part of this milestone.
- AI ranking or semantic selection (M2), multiple transcripts/assets (M3), persistence, network, telemetry.
- Re-parsing or re-validating the transcript — M1.1 consumes the already-canonical `TranscriptDocumentV1`.
- New runtime dependencies; matching uses only the standard library and existing pure packages.

## Contracts and invariants

**Inputs (pure):** the immutable `TranscriptDocumentV1` from `transcript`, plus an ordered list of passage strings. No `process`, filesystem, or framework imports (enforced by the boundary guard).

**Output invariants:**

- Every input passage produces exactly one result entry, in input order; nothing is silently dropped.
- Every emitted `SourceRange` is a valid `domain` `SourceRange` (positive duration) and lies within the transcript's covered range — constructed via `createSourceRange`, never hand-built.
- A passage spanning entries `[i..j]` yields `start = entry[i].startUs`, `end = entry[j].endUs`, with `i ≤ j` and no reordering of entries.
- Matching is a pure function of (normalized passages, transcript): identical inputs give byte-identical results; no wall-clock, randomness, or locale-dependent behavior. Normalization is idempotent and documented.
- Confidence and match kind (`exact` | `fuzzy`) are explicit; fuzzy matches carry the tolerance that admitted them.

## Failure and edge modes (each has a defined, tested result)

- Passage not present at all → `unmatched` with a reason code (e.g. `NO_MATCH`).
- Passage present in multiple locations → `ambiguous` with all candidate ranges in transcript order.
- Passage present but only within fuzzy tolerance → `matched` kind `fuzzy` with reduced confidence.
- Empty passage / whitespace-only passage → `unmatched` with a distinct reason; never a zero-length range.
- Passage contiguous in words but spanning entries with a timing gap (a natural pause) → `matched`; the `SourceRange` spans first→last entry and **includes the gap** (decision 3). A passage that would require skipping transcript words is not one contiguous selection → `unmatched`.
- Transcript with zero entries → all passages `unmatched`.

## Acceptance criteria

- `pnpm check:boundaries` proves the new package is framework-independent and only imports `domain` + `transcript`.
- Unit tests cover exact, fuzzy, ambiguous, unmatched, empty, and boundary cases from a fixture transcript (text-only; no binary media).
- Property-based tests hold the output invariants above across generated transcripts and passages.
- The full authoritative sequence is green in hosted CI on the candidate SHA.
- No new dependency, no I/O, no renderer/IPC change.

## Resolved decisions (approved 2026-07-22)

1. **Package name: `selection`** — named for the domain artifact (a resolved selection) that downstream milestones consume, not the matching algorithm.
2. **Fuzzy model: token-level, exact-first.** Normalize to tokens; a passage matches when its token sequence appears as a contiguous run of transcript tokens. Exact matching is attempted first; fuzzy is a fallback using **bounded token edit distance** (not character-level Levenshtein). Confidence = `1 − (token errors ÷ passage tokens)`. Starting threshold: a fuzzy match is admitted only when token errors ≤ `min(ceil(0.15 × passageTokens), 3)`. This threshold is a documented, tunable constant to be revisited against real transcript data.
3. **Gap handling: bridge time, reject token gaps.** A contiguous-in-words passage yields a range that spans from the first entry's start to the last entry's end, including any intra-span timing gaps. A passage requiring skipped transcript words is `unmatched`.
4. **Diacritics: fold in the match key only.** Fold accents (`á→a`, `ç→c`, …) when building the normalized comparison key; the transcript's original text and timestamps are never altered — folding affects comparison, not output.
