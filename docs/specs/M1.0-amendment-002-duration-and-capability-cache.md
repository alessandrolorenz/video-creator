## Exact duration-string grammar

Replace “strict finite positive base-10 decimal syntax” in Amendment 001 with the following complete rule.

A duration candidate:

1. must be a string containing 1–128 UTF-16 code units;
2. must match `^(?:0|[1-9][0-9]*)(?:\.[0-9]+)?$` exactly;
3. is not trimmed or otherwise normalized before matching;
4. is converted with `Number(candidate)` only after matching;
5. must convert to a finite number greater than zero;
6. is converted exactly once with `Math.round(seconds * 1_000_000)`;
7. must produce a positive JavaScript safe integer.

Accepted lexical examples include `1`, `0.5`, `1.0`, and `10.000000`.

Rejected lexical examples include the empty string, `+1`, `-1`, `01`, `.5`, `1.`, `1e3`, ` 1`, `1 `, `NaN`, and `Infinity`. Syntactically valid `0` or a positive fraction that rounds to zero microseconds is rejected by the positivity rule.

Frozen numeric boundary examples under the required JavaScript conversion are:

- `0.0000004` → `0` microseconds → unavailable;
- `0.0000005` → `1` microsecond → valid;
- `9007199254.740990` → `9_007_199_254_740_990` microseconds → valid safe integer;
- `9007199254.740991` → an unsafe rounded JavaScript integer → unavailable.

Classification:

- an absent duration member or a string that fails any duration rule is an unavailable duration candidate and participates in the frozen fallback algorithm;
- a present duration member whose value is not a string is a structurally invalid field and produces `PROBE_OUTPUT_INVALID` at the existing media-structure validation stage;
- candidate unavailability alone never produces `PROBE_OUTPUT_INVALID`;
- if every permitted candidate is unavailable, the existing result is `MEDIA_DURATION_INVALID`;
- `DURATION_FALLBACK_USED` is emitted only when the greatest valid selected-stream candidate is used because `format.duration` was unavailable.

## Exact capability-cache behavior

Clarify the Amendment 001 capability cache as follows:

1. The cache is private to one live utility-worker instance and starts empty when that worker is created.
2. Its key uses exact JavaScript string/code-unit equality on the already-resolved executable string; no path normalization, canonicalization, case folding, or filesystem identity is applied.
3. Only capability PASS is inserted.
4. A cache hit returns the previously sanitized version line without spawning a second `-version` process.
5. Every failure or cancellation leaves the key absent, so the next operation using the same exact string invokes `-version` again.
6. A successful retry inserts the PASS; the following same-key operation is then a cache hit.
7. A worker exit or replacement discards its cache. A recreated worker begins empty and performs the check again. No cache is persisted in main, renderer, disk, or across application sessions.
