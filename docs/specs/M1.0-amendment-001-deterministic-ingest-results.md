# M1.0 — Amendment 001: Deterministic Ingest Results and Ownership

Status: APPROVED / NORMATIVE — applied alongside the frozen M1.0 specification; implementation not authorized

Prepared: 2026-07-22 (America/Sao_Paulo)

## Purpose

Close every implementation-time ambiguity identified by the independent plan judge without changing M1.0 product scope. This amendment defines:

- exact result/error conditions and precedence;
- exact rational, duration, VFR, rotation, and warning algorithms;
- an exact `ffprobe -version` capability contract;
- main-process ownership of executable configuration;
- required real utility-process integration evidence using a fake executable;
- checkpoint-separation constraints.

No ingest capability is added beyond the frozen M1.0 objective.

## Old rules and gaps

The frozen specification currently:

- lists closed result/error codes without a complete condition/precedence matrix;
- says missing/invalid rational values produce warnings or errors without deciding which;
- names VFR and rotation warnings without exact triggers;
- names `FFPROBE_INCOMPATIBLE` without a compatibility algorithm;
- says executable configuration is main-process-only but does not fully assign its lifecycle;
- requires tests for every closed error/UX state but does not explicitly require a compiled-worker/real-utility-process integration;
- permits a plan shape that grouped too many main/IPC/preload responsibilities into one checkpoint.

Those gaps made deterministic tests impossible for parts of the plan.

## New result codes

Add two failure codes to the frozen closed set:

- `INVALID_REQUEST` — a renderer-to-main request fails its strict versioned schema before work; an invalid privileged main/worker message follows the separate `INTERNAL_ERROR` rule below;
- `FFPROBE_CONFIGURATION_INVALID` — the optional privileged development override exists but violates its frozen absolute-path contract.

The complete V1 closed set is the original list plus these two codes. No implementation may invent another renderer-visible code without a later approved amendment.

## Global terminal-result rule

Every dialog/import/probe operation has one opaque job ID and exactly one terminal latch.

- JavaScript event-loop delivery order is authoritative.
- The first validated terminal event that calls the latch wins.
- Cancellation latches its cancellation result before issuing abort/kill.
- Timeout latches before issuing abort/kill.
- An output-limit breach latches before issuing abort/kill.
- Worker exit/disconnect latches only when no typed terminal result has already latched.
- Later close, error, message, timeout, cancellation, or output events are ignored except for privileged diagnostics.
- No terminal result may be replaced by a later, apparently more specific condition.
- `INTERNAL_ERROR` is last-resort handling for an unexpected privileged invariant/exception only; it is never used for a listed input, file, process, metadata, worker, or transcript condition.

This rule resolves asynchronous races without depending on wall-clock timestamp comparison.

## Request and worker precedence

The following checks occur before any dialog, filesystem, utility-process, or subprocess work:

| Order | Condition | Result |
|---:|---|---|
| 1 | Renderer/main request fails strict schema/version/extra-key validation | `INVALID_REQUEST` |
| 2 | Transcript import has no exact active ready asset or supplied opaque asset ID does not equal it | `TRANSCRIPT_PREREQUISITE_MISSING` |
| 3 | Main/worker message fails strict schema/version/job identity validation | `INTERNAL_ERROR` to renderer; privileged protocol diagnostic retained |
| 4 | Utility process cannot be forked, connected, or becomes unavailable before accepting the typed job | `WORKER_UNAVAILABLE` |

A stale but well-formed result for a replaced/cancelled job is ignored; it is not renderer-visible and does not create a new code.

## Native dialog and file precedence

For both media and transcript selection:

| Order | Condition | Result |
|---:|---|---|
| 1 | User cancels/closes the native dialog or selects no file | `DIALOG_CANCELLED` outcome |
| 2 | Selected path cannot be canonicalized, stated, or opened because it is missing, inaccessible, or a filesystem operation fails | `FILE_UNAVAILABLE` |
| 3 | Canonical selected target is not a regular file | `FILE_NOT_REGULAR` |

Additional media check:

| Order | Condition | Result |
|---:|---|---|
| 4 | Media regular-file size is exactly zero bytes | `MEDIA_EMPTY` |

Additional transcript check:

| Order | Condition | Result |
|---:|---|---|
| 4 | Raw transcript size is greater than 20 MiB | `TRANSCRIPT_TOO_LARGE` |
| 5 | Cancellation latches before bounded read/decode/parse completes | `TRANSCRIPT_CANCELLED` outcome |

File extensions and dialog filters never determine success or an error code.

## Main-owned executable configuration

Executable configuration is resolved once by Electron main when the ingest service is constructed:

1. Read optional `AI_VIDEO_ASSEMBLY_FFPROBE_PATH` in main only.
2. If absent, resolve to the bare executable name `ffprobe`.
3. If present, the raw value must be 1–4,096 UTF-16 code units, contain no NUL, and be an absolute path according to the current platform.
4. Do not trim, expand `~`, interpolate variables, search directories, canonicalize a nonexistent executable, or accept a renderer value.
5. Violation of step 3 returns `FFPROBE_CONFIGURATION_INVALID` before utility-process creation.
6. A valid resolved string is passed to the utility worker through a strict privileged configuration message.
7. The utility worker never reads `process.env`, chooses an executable, or changes the resolved string.
8. Renderer snapshots expose only capability state and a sanitized version line; never the resolved string or whether the override was used.

If a syntactically valid absolute override or bare `ffprobe` cannot be spawned with OS error `ENOENT`, the result is `FFPROBE_NOT_FOUND`.

## `ffprobe -version` compatibility contract

The worker performs a capability check once per resolved executable string per application session before its first media probe. Successful capability state may be cached; a failure is not cached and may be retried by a later user action.

Invocation is exactly:

- executable: the already-resolved privileged string;
- arguments: `['-version']`;
- `shell: false`;
- stdin: ignored;
- stdout/stderr: separate bounded pipes;
- timeout: 5 seconds;
- stdout cap: 64 KiB;
- stderr cap: 64 KiB.

Result mapping:

| Condition | Result |
|---|---|
| Spawn emits `ENOENT` | `FFPROBE_NOT_FOUND` |
| Spawn emits another OS error or terminates by signal before cancellation | `FFPROBE_INCOMPATIBLE` |
| Timeout, stdout/stderr cap breach, nonzero exit, invalid UTF-8 stdout, or no compatible first line | `FFPROBE_INCOMPATIBLE` |
| Cancellation wins the terminal latch | caller operation's cancellation outcome |
| Exit 0 and compatible first line | capability PASS |

The compatible line is the first non-empty LF/CRLF-normalized stdout line. It must:

- contain 1–256 UTF-16 code units;
- contain no C0/C1 control character;
- match `^ffprobe version [^\s]+(?: .*)?$`.

The sanitized renderer-visible version is exactly that validated line. Raw stdout/stderr never crosses the privileged boundary. No semantic-version minimum is claimed in M1.0; compatibility means successful identity/capability for the frozen probe contract.

## Media-probe result precedence

After capability PASS, media probing uses the frozen fixed arguments and caps. The terminal mapping is:

| Order / terminal condition | Result |
|---|---|
| Cancellation wins latch | `PROBE_CANCELLED` outcome |
| 30-second timer wins latch | `PROBE_TIMEOUT` |
| stdout or stderr would exceed its frozen cap | `PROBE_OUTPUT_LIMIT` |
| Spawn emits `ENOENT` | `FFPROBE_NOT_FOUND` |
| Spawn emits another OS error, or child ends by unrequested signal | `PROBE_FAILED` |
| Child closes with nonzero numeric exit code | `MEDIA_UNSUPPORTED` |
| Exit 0 but stdout is invalid UTF-8, invalid JSON, wrong top-level/stream/format structure, or contains semantically invalid required field types | `PROBE_OUTPUT_INVALID` |
| Parsed streams contain no eligible primary video | `VIDEO_STREAM_MISSING` |
| Parsed streams contain no primary audio | `AUDIO_STREAM_MISSING` |
| Duration remains unavailable/invalid after the frozen fallback | `MEDIA_DURATION_INVALID` |
| All checks pass | success |

`PROBE_FAILED` is process/runtime failure. `MEDIA_UNSUPPORTED` is a normal `ffprobe` nonzero exit for the selected regular non-empty file. Raw/localized stderr is never parsed to choose a code.

## Rational parsing

For each primary-video `avg_frame_rate`, `r_frame_rate`, and `time_base`:

1. Accept only a string matching `^[0-9]+/[0-9]+$`.
2. Parse numerator and denominator as base-10 `BigInt`.
3. Both values must be greater than zero.
4. Reduce using greatest common divisor.
5. Reduced numerator and denominator must each be at most `Number.MAX_SAFE_INTEGER`.
6. A value failing any step is unavailable, not a hard ingest error.

The renderer-safe summary fields are optional rationals. Missing/invalid rate/time-base metadata never causes `PROBE_OUTPUT_INVALID` or ingest failure when the containing field type itself is permitted by the captured output shape.

## Duration algorithm

Candidate duration strings use strict finite positive base-10 decimal syntax, convert with the frozen seconds-to-microseconds rule, and must produce positive safe-integer `TimeUs`.

1. If `format.duration` is valid, use it and do not emit `DURATION_FALLBACK_USED`.
2. Otherwise evaluate duration only from the selected primary video and primary audio streams.
3. Use the greatest valid positive stream duration.
4. If step 3 succeeds, emit `DURATION_FALLBACK_USED`.
5. If no candidate succeeds, return `MEDIA_DURATION_INVALID`.

Duration disagreement does not choose another candidate or create another warning in M1.0.

## Warning algorithms and order

Warnings are deduplicated and returned only in this fixed order:

1. `DURATION_FALLBACK_USED`;
2. `VARIABLE_FRAME_RATE_SUSPECTED`;
3. `RATE_METADATA_MISSING`;
4. `ROTATION_METADATA_PRESENT`.

Exact triggers:

- `DURATION_FALLBACK_USED`: and only when the duration fallback algorithm selects a stream duration.
- `RATE_METADATA_MISSING`: at least one of primary-video `avg_frame_rate`, `r_frame_rate`, or `time_base` is unavailable after frozen rational parsing.
- `VARIABLE_FRAME_RATE_SUSPECTED`: both average and real/base frame rates are available and their reduced numerator/denominator pairs are unequal. Missing rates emit only `RATE_METADATA_MISSING`.
- `ROTATION_METADATA_PRESENT`: selected display rotation normalizes to a nonzero integer degree.

Rotation selection/normalization:

1. Prefer the first finite integer `rotation` value in primary-video `side_data_list` order.
2. Otherwise use a finite integer decimal string from primary-video tag `rotate`.
3. Ignore invalid/non-integer candidates.
4. Normalize the selected integer with `((value % 360) + 360) % 360`, yielding 0–359.
5. Store the normalized integer when selected; emit the warning only when it is nonzero.

## Transcript result precedence

After file checks and successful fatal UTF-8 decoding:

| Order | Condition | Result |
|---:|---|---|
| 1 | Fatal UTF-8 decoding fails or decoded text contains a replacement character | `TRANSCRIPT_ENCODING_INVALID` |
| 2 | JSON syntax fails, root is not exactly one object, or trailing non-whitespace data exists | `TRANSCRIPT_JSON_INVALID` |
| 3 | `schemaVersion` is absent, non-integer, or not literal `1` | `TRANSCRIPT_SCHEMA_UNSUPPORTED` |
| 4 | Entry count, total text, or frozen string/object limits are exceeded | `TRANSCRIPT_LIMIT_EXCEEDED` |
| 5 | Other root/entry shape, extra-key, type, confidence, text, numeric, conversion, or interval validity fails | `TRANSCRIPT_ENTRY_INVALID` |
| 6 | During adjacent scan, `current.startUs < previous.startUs` | `TRANSCRIPT_ORDER_INVALID` |
| 7 | Otherwise, during adjacent scan, `current.startUs < previous.endUs` | `TRANSCRIPT_OVERLAP_UNSUPPORTED` |
| 8 | Any valid converted entry ends after active asset duration | `TRANSCRIPT_OUT_OF_BOUNDS` |
| 9 | All checks pass | success |

The adjacent scan stops at the first failing pair. Order is checked before overlap for that pair. No sorting, merging, clamping, or repair occurs.

## Worker availability and internal failure

- `WORKER_UNAVAILABLE`: utility-process fork throws/fails, required message channel cannot be established, or the utility process exits/disconnects before a typed terminal result for an accepted active job.
- `INTERNAL_ERROR`: unexpected main/worker invariant or uncategorized implementation exception after all explicit mappings above. Renderer message is fixed and path/content-free; diagnostics remain privileged.

Worker exit after a typed terminal latch does not change the result.

## Required test and integration changes

The corrected plan must map every result/warning to owner, checkpoint, positive test, negative test, and precedence/race test where applicable.

It must also require an integration that:

- builds the actual utility-worker artifact;
- launches it through real Electron `utilityProcess` from a test harness;
- supplies a temporary text JavaScript fake executable or Node executable/fixture outside tracked media;
- exercises version PASS/failure, one probe success/failure, cancellation, path redaction, and shutdown;
- uses no binary media, real `ffprobe`, registry dependency, shell, or public path logging.

## Checkpoint-separation rule

The corrected plan must separate at least:

1. pure transcript/domain;
2. pure media parser/contracts;
3. worker protocol/runner/build;
4. main-owned executable configuration, dialogs, bounded read, utility client, and ingest controller;
5. strict IPC handlers, preload, lifecycle/shutdown, and real utility-process integration;
6. renderer UX;
7. docs and clean-room evidence.

Each checkpoint receives separate authorization, tests first, one local commit, complete verification, precise rollback, and a stop.
